# Jason Yuan - Speculative Futures and Fluid Intent

Designer of Mercury OS (2019), the most influential speculative operating-system concept of the decade, and co-founder of new.computer (Dot), an early serious attempt at AI-native personal software. Yuan's value to this skill is a disciplined method for imagining beyond the current paradigm: he questions the app, the window, and the file as units of computing, and replaces them with intention as the organizing primitive.

## Core principles, translated to desktop UI

1. **Organize around intention, not application.** Mercury OS's core move: users do not want "open mail app, open calendar app, open docs app"; they want "prepare for tomorrow's meeting." A future-facing desktop UI assembles the relevant capabilities and content around the stated or inferred intention, in a single fluid space. Practical near-term version: cross-cutting workspaces or command flows that pull actions from many domains into one thread.
2. **Content over chrome, radically.** Mercury removed docks, desktops, and window management entirely: only the content of the current flow, with everything else summoned on demand. Near-term translation: aggressive progressive disclosure; the resting state of the app shows the work and almost nothing else, and controls materialize in context.
3. **Design for focus as a humane obligation.** Yuan framed Mercury around excluded users (ADHD, anxiety) failed by fragmented, notification-driven UI. One thing asks for attention at a time; interruptions are queued into the flow rather than stacked on top of it. This reframes "minimalism" from aesthetic to accessibility.
4. **Conversation is an ingredient, not the interface.** From the new.computer era: pure chat is a regression for most tasks (low information density, no direct manipulation), but language is unmatched for expressing intent. The synthesis: language in, rich manipulable artifacts out. An AI-native desktop tool should accept a sentence and respond with an editable object (a table, a plan, a canvas), never with a paragraph the user must parse.
5. **Speculate with rigor.** Mercury OS worked as provocation because it had a coherent internal logic: defined primitives (modules, flows, spaces), rules for how they compose, and honest acknowledgment of what was unsolved. When imagining future UI, define the primitives and their grammar first; a beautiful frame with no system is a mood board.

## Signature moves to steal

- **The intent field.** A single, always-available input that accepts natural language and resolves to structured action across the whole product (superset of the command palette: verbs plus objects plus parameters in one utterance). Pair with visible, editable interpretation so the system's understanding is inspectable.
- **Flows as first-class objects.** Persistent, resumable sequences of connected content ("everything for the board meeting") that the user can name, revisit, and share, cutting across the app's nouns.
- **Ambient status, summoned detail.** Peripheral information rendered as calm ambient signal (tone, small glyphs) with full detail one gesture away, instead of badge counts and notification trays.
- **Generative UI within guardrails.** Let the system compose views from a constrained kit of vetted components in response to intent, so novel screens still obey the design system.

## Never copy

- Mercury's visual language (the fog gradients, ethereal type, serif headers). It was widely admired and is instantly recognizable; lifting it is costume, not thinking.
- Chrome-removal beyond user control. Mercury was a concept; shipped software needs escape hatches, discoverability, and expert affordances. Radical focus must be reversible.

## Interrogation questions

What is the user's intention, and what would the UI look like if that, not the app's features, organized the screen? Where could a sentence replace a five-step flow, and what manipulable artifact should come back? What are the primitives of this future concept, and what is their grammar?
